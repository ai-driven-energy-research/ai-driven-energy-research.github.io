I'm starting an academic journal called AI-Driven Energy Research (AIDER), and I want you to be part of the founding editorial board.

Before you close this tab — this is not a traditional journal. There's no publisher behind it, no fees, no bureaucracy. It's a group of PhD students building something new on GitHub. Running costs are kept minimal and covered independently — not by readers or authors.

The Problem

You've read papers where the results can't be reproduced. You've tried to find the code — it doesn't exist. You've emailed the authors — no reply. The paper is cited 200 times but nobody can actually use it.

Meanwhile, we're all using AI to help with our research — writing code, running experiments, even drafting papers. But we're hiding it, because journals haven't caught up.

What AIDER Does Differently

Every paper must be fully reproducible. Code, data, and a script that regenerates every figure. An automated CI pipeline verifies this before human review even begins. If the code doesn't run, the paper is rejected.
Every paper must show how it was made. If you used AI agents to help write or code — share the logs. If you made a key decision — document it. The research process is part of the contribution.
AI is encouraged, not hidden. We think AI-driven research is the future. AIDER is built for researchers who actually work this way.
Review is automated where possible. AI agents assess methodology and code quality. Automated pipelines verify reproducibility. Human editors make final decisions. The entire review process is public.
Free for readers. Free for authors. Always. No APCs, no subscriptions. Operating costs are covered by supporters who believe in open science — never by those who read or publish here.

How It Works — The Full Review Process

Below is the complete journey of a paper, from submission to publication. As an editor, you only need to worry about Steps 4–7 — everything before that is automated or handled by the author. But we're showing you the whole picture so you understand the context.

1
Author
Author Submits a Paper
The author creates a GitHub repository with their paper, code, data, and a process log. They then open a submission issue on our submissions page.
2
Automated
Reproducibility Check (CI)
An automated pipeline clones the author's repo, installs dependencies, and runs their code. It checks: Does the code run? Does it produce the claimed results? This happens automatically — you don't need to do anything.
3
Automated
AI Reviewer Agent
An AI agent reads the paper and generates a preliminary review — checking structure, methodology, and flagging potential issues. This report will be posted as a comment on the submission issue for you to read.
4
You
You Get Notified
The editor-in-chief assigns you to review a paper. You'll receive a GitHub notification email with a link to the submission issue. The issue page contains: the paper link, the author's repo, the CI results, and the AI review.
5
You
Read the Paper & Review Materials
Here's what to do, step by step:
  1. Open the submission issue — click the link in the email you received. This is a page on GitHub that looks like a forum thread.
  2. Read the AI review — scroll down to the comments. The AI agent will have posted its assessment. This gives you a head start — it highlights strengths, weaknesses, and things to look out for.
  3. Check the CI results — near the top of the issue, you'll see a green checkmark (code ran successfully) or a red X (code failed). If it's a red X, the paper already has a reproducibility problem.
  4. Read the paper — the submission issue contains a link to the author's repository. Go there and open the PDF in the paper/ folder. Read it like you would any journal paper.
  5. Browse the code and data — look through the code/ and data/ folders. You don't need to understand every line, but check that it looks reasonable and matches what the paper claims.
  6. Check the process log — open the process-log/ folder. This documents how the research was done, including any AI tools used. Is it detailed enough that someone could follow the same path?
6
You
Try Running the Code (Optional but Encouraged)
The CI already ran the code automatically, but if you want to verify it yourself:
  1. Open a terminal on your computer
  2. Type: git clone <repo-url> (copy the URL from the author's GitHub page — click the green "Code" button)
  3. Follow the instructions in code/README.md to install dependencies
  4. Run: bash results/reproduce.sh
  5. Check if the generated figures match the ones in the paper
New to Git? Don't worry — you can also just browse the code on GitHub in your web browser without cloning anything. The CI results already tell you whether the code runs.
7
You
Post Your Review
Go back to the submission issue on GitHub and write a comment with your review. Use this structure:
## Review — [Your Name]

### Summary
2-3 sentences: what is this paper about, and what's your overall impression?

### Research Quality
- Is the contribution novel?
- Is the methodology sound?
- Do the results support the claims?
- Is it well-written?

### Reproducibility
- Did the CI pass? (green checkmark = yes)
- If you ran the code yourself, did it work?
- Do the outputs match the paper?

### Process Log
Is the research process well-documented?

### Recommendation
Pick one: Accept / Minor revision / Major revision / Reject

### AI Disclosure
Did you use any AI tools to help with this review? If so, which ones?
That's it — just a comment on the issue. No forms, no separate system, no login to a review portal.
8
Editor-in-Chief
Decision & Publication
The editor-in-chief reads all reviews and makes a final decision. If accepted, the paper is archived in the AIDER GitHub organisation and assigned a DOI via Zenodo. The paper is now permanently published and citable.

New to GitHub?

You don't need to be a GitHub expert. Here's the minimum you need:

  1. Create a free account at github.com/signup — use your academic email
  2. Tell us your username so we can add you to the AIDER organisation — this lets you get assigned to reviews
  3. Turn on email notifications — go to Settings → Notifications and make sure "Email" is checked. This is how you'll know when a paper is assigned to you.

That's all the setup. When it's time to review, everything happens on a single GitHub page (the submission issue). You read, you comment — same as leaving a reply on any website.

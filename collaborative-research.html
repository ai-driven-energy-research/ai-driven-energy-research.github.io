TL;DR

  • One person forks the paper template — everyone else clones it
  • Each person works on their own branch, then opens a pull request to merge
  • Each person keeps their own AI session logs in a personal folder
  • Before submitting, merge everything to main, run reproduce.sh, and one person opens the submission issue

Setting Up a Team Repository

One team member (the "repo owner") creates the shared repository. Everyone else clones it and works from there.

  1. Fork the template. The repo owner goes to the AIDER paper template and clicks "Use this template" to create a new repository.
  2. Add collaborators. Go to Settings → Collaborators → Add people. Add each team member by their GitHub username. They'll receive an invitation email.
  3. Agree on branches. Decide who works on what. A simple convention: one branch per person or per task (e.g., alice/methodology, bob/data-pipeline). The main branch is for finished, merged work only.
  4. Set up branch protection (optional). Go to Settings → Branches → Add rule for main. Require pull requests before merging. This prevents accidental pushes to main.

Branch-and-Merge Workflow

If you've never used Git branches before, here's the idea: everyone works on their own copy of the code, then merges it back together when ready. This prevents stepping on each other's work.

1
Each team member
Create a branch
Each person creates a branch for their work. This is a personal copy where you can make changes without affecting anyone else.
git checkout -b alice/methodology
2
Each team member
Do your work
Write code, run experiments, draft your section of the paper. Commit your changes regularly with descriptive messages.
git add . then git commit -m "Add wind speed preprocessing"
3
Each team member
Push your branch
Upload your branch to GitHub so the team can see your work.
git push origin alice/methodology
4
Each team member
Open a pull request
On GitHub, click "Compare & pull request" for your branch. Describe what you changed and why. This is where your team reviews each other's work before merging.
5
Team review
Review and discuss
Other team members review the pull request. Leave comments, suggest changes, ask questions. When everyone is happy, approve the PR.
6
Merge
Merge to main
Click "Merge pull request" on GitHub. Your work is now part of the shared main branch. Delete your feature branch to keep things tidy.

Multiple People + Multiple AI Agents

In collaborative AIDER research, each person may use a different AI agent (or the same one in different sessions). The key rule: every person's AI sessions must be logged separately and attributable.

Scenario How It Works Process Log Approach
Each person uses their own agent Alice uses Claude Code, Bob uses Aider, Carol uses Codex CLI. Each works on their own branch with their own agent. Each person saves logs in process-log/ai-sessions/alice/, .../bob/, etc. Each folder contains that person's full session transcripts.
Pair programming with AI Two people sit together (or screen-share) and use one AI agent interactively. One drives, one reviews. Save the session in a shared folder like process-log/ai-sessions/pair-alice-bob/. Note in the log who was driving and who was reviewing.
Shared API pipeline The team builds a shared script that calls an AI API (e.g., for batch analysis). The script runs programmatically. The script itself lives in code/. API request/response logs go in process-log/ai-sessions/pipeline/. Document the pipeline design in the process log README.
Sequential handoff Person A does data collection, hands off to Person B for modelling, hands off to Person C for analysis. Each uses their own AI agent for their stage. Each person logs their stage separately. The process log README describes the handoff sequence and what each person was responsible for.

Team Process Log Structure

For team submissions, the process log has a slightly expanded structure. Each person gets their own folder for AI sessions, and there's a shared decisions log:

process-log/
├── README.md # Narrative overview: who did what, workflow summary
├── contributions.md # Who contributed what (like a CRediT statement)
├── shared-decisions/ # Decisions made by the team together
│ └── decisions.md # Timestamped log of group decisions
├── ai-sessions/
│ ├── alice/
│ │ ├── session-01.jsonl # Alice's first AI agent session
│ │ ├── session-02.jsonl # Alice's second session
│ │ └── README.md # What Alice used: tool, version, purpose
│ ├── bob/
│ │ ├── session-01.txt # Bob used a different agent/format
│ │ └── README.md
│ └── pipeline/
│ ├── batch-run-01.json # Shared API pipeline logs
│ └── README.md
└── human-decisions/
├── alice-decisions.md # Alice's individual research decisions
└── bob-decisions.md # Bob's individual research decisions

contributions.md

This file maps each person to their contributions, similar to a CRediT author statement. Example:

Alice Chen Conceptualization, Methodology, Software (data preprocessing), Writing (Introduction, Methodology). AI tools: Claude Code for all coding sessions.
Bob Kumar Data curation, Software (model training), Visualization, Writing (Results). AI tools: Aider with GPT-4 for model development.

Handling Merge Conflicts

A merge conflict happens when two people edit the same part of the same file. Git doesn't know which version to keep, so it asks you to choose.

What it looks like

When you try to merge and there's a conflict, Git marks the file like this:

<<<<<<< HEAD
learning_rate = 0.001
=======
learning_rate = 0.01
>>>>>>> bob/hyperparameters

The top section is your version, the bottom is the other person's version. You need to pick one (or combine them) and delete the conflict markers.

How to resolve

  1. Open the file with conflicts
  2. Find the <<<<<<< markers
  3. Decide which version to keep (or write a new version that combines both)
  4. Delete the conflict markers (<<<<<<<, =======, >>>>>>>)
  5. Save the file, then git add and git commit

How to prevent conflicts

  • Work on different files. If Alice works on preprocessing.py and Bob works on model.py, they'll never conflict.
  • Merge often. Don't let branches diverge for weeks. Merge to main frequently and pull the latest main into your branch.
  • Communicate. If two people need to edit the same file, coordinate who goes first.

Submitting as a Team

When the research is complete and ready for submission:

  1. Merge all branches to main. Every team member's work should be merged via pull requests. No open feature branches left.
  2. Fill in contributions.md. Map each author to their contributions and AI tools used.
  3. Run reproduce.sh. One person clones a fresh copy of the repo and runs bash results/reproduce.sh. This simulates what the AIDER CI will do. Everything must pass.
  4. Complete the reproducibility checklist. Fill in REPRODUCIBILITY.md together.
  5. One person opens the submission issue. Go to the submissions page and open a new issue. List all authors and link to the repository.
Who opens the submission? It doesn't matter which team member opens the submission issue. Typically the corresponding author does it. All authors should be listed in the issue and in the repository's README.

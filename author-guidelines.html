Overview

AIDER accepts original research at the intersection of artificial intelligence and energy systems. Every submission must be a complete, executable research package.

Manuscript Preparation

Format

Manuscripts must be submitted as source files (LaTeX or Markdown), not compiled PDFs. Place all manuscript files in the paper/ directory of your repository. There is no strict page limit, but clarity and conciseness are expected.

Structure

Papers should follow standard academic structure: Abstract, Introduction, Methodology, Results, Discussion, Conclusion, References. Variations are acceptable where the research warrants it.

Language

All submissions must be in English. AI-assisted writing is permitted and must be disclosed in the process log.

Repository Requirements

Start by creating a repository from the AIDER paper template. Your repository must contain:

Code (code/)

  • All source code required to reproduce every result in the paper
  • A README.md with clear setup and execution instructions
  • Dependency management via requirements.txt, environment.yml, pyproject.toml, or Dockerfile
  • Code must execute without errors on a clean environment

Data (data/)

  • All data necessary to reproduce results
  • If data exceeds GitHub's file size limits, host on Zenodo or HuggingFace Datasets with a persistent DOI
  • A README.md describing data format, source, collection method, and license
  • Synthetic or generated data must include the generation script

Process Log (process-log/)

This is what makes AIDER unique. You must document how the paper was produced:

  • AI sessions (process-log/ai-sessions/): Full transcripts or logs of any AI agent sessions used during research — including prompts, outputs, and tool use. Accepted formats: text logs, JSON exports, JSONL conversation dumps, or screenshots with descriptions
  • Human decisions (process-log/human-decisions/): A timestamped log of significant human decisions — parameter choices, methodology pivots, manual corrections, data cleaning decisions. Can be a simple Markdown file with dated entries
  • Overview (process-log/README.md): A narrative summary of the research workflow — which stages used AI, which were manual, and how human and AI contributions interacted

Reproducibility Script (results/reproduce.sh)

  • A single script that regenerates all key figures and tables from the paper
  • Must exit with code 0 on success
  • Random seeds must be fixed for deterministic output
  • Execution time should be documented in the script header (the CI allows up to 60 minutes)

Reproducibility Checklist (REPRODUCIBILITY.md)

Complete every item in the provided checklist before submitting. Incomplete checklists will result in desk rejection.

Openness Tiers & Reproducibility

We recognise that not all research can be fully open. A month-long CFD simulation may produce terabytes of data that cannot be uploaded. An industry partner may restrict access to operational datasets. These are real constraints, and we don't want to exclude valuable research because of them.

AIDER defines four openness tiers based on what the author can share:

Tier Code Data Process Log AI Verification Review Process
Tier 1: Fully Open ✓ All public ✓ All public ✓ Full AI sessions & human decisions ✓ End-to-end Currently Supported
AI agents verify everything. Editor reviews AI reports.
Tier 2: Open Code, Large Data ✓ All public ~ Reduced or hosted externally ✓ Full ~ Partial Under Development
AI verifies code and post-processing. Full simulation not re-runnable.
Tier 3: Open Code, Restricted Data ✓ All public ✗ Restricted (sample or synthetic provided) ~ Redacted where proprietary data appears in prompts ~ Limited Under Development
AI verifies code on sample data. Human reviewer needed for full assessment.
Tier 4: Partially Open ~ Partial (e.g., commercial solver) ✗ Restricted ✗ Summary only (methods narrative without full logs) ✗ Very limited Not Yet Supported
Requires multiple expert human reviewers.

What we currently accept

Tier 1 submissions are our focus right now. This is what AIDER was built for — fully open, fully reproducible research where AI agents can verify everything end-to-end. It's our differentiator, and it's where we can guarantee the highest review quality with our small editorial team.

We are actively working towards supporting Tiers 2 and 3 as the journal grows and we build a larger pool of domain-expert reviewers. If your research falls into these tiers, we still welcome you to get in touch — we may be able to accommodate your submission on a case-by-case basis, especially if you can suggest qualified reviewers in your field.

Guidelines for Tier 2 and 3 submissions

If your research involves large datasets or restricted data, here is how to maximise what can be verified:

  • Large simulation data (CFD, climate, etc.): Provide all input files (mesh scripts, solver configuration, boundary conditions), post-processing scripts, and reduced/processed data (time-averaged fields, statistics). Host large files on Zenodo (up to 50 GB per record). Include SHA-256 checksums of the full dataset so others with access can verify integrity.
  • Proprietary or sensitive data: Provide a synthetic or anonymised sample dataset that allows the code to run and demonstrates the methodology. Include a data access statement describing what the data is, where it is held, and how other researchers can request access.
  • Commercial solvers: Provide all input/configuration files so that anyone with a licence can reproduce the simulation. Document the exact solver version and settings used.

The guiding principle is: publish the maximum you can, and be transparent about what you can't.

Reproducibility badges

Every published paper receives a reproducibility badge based on what the AI agents and reviewers could verify:

  • Fully Reproducible — AI agents ran everything end-to-end and all results were verified
  • Partially Reproducible — code and post-processing verified, but full simulation or data generation could not be re-run (with a clear explanation of what was and wasn't checked)
  • Methodology Verified — code logic and methodology reviewed, but results depend on data that could not be independently verified

These badges are assigned honestly based on what was actually checked — not on the quality of the research itself. A Tier 2 paper with a "Partially Reproducible" badge can be excellent science. The badge simply tells the reader what level of independent verification was performed.

A note to authors: We are a young journal figuring out the best way to handle the full spectrum of research openness. The policies above for Tiers 2–4 are evolving. If you have research that doesn't fit neatly into Tier 1 but you believe belongs in AIDER, please reach out. We would rather work with you to find a solution than turn away good science. This will become easier as the journal grows and our reviewer network expands.

AI Disclosure

AIDER encourages the use of AI tools throughout the research process. All AI use must be documented:

  • Name and version of each AI tool used (e.g., Claude 3.5 Sonnet, GPT-4, GitHub Copilot)
  • How each tool was used (writing, coding, analysis, literature review, etc.)
  • Session logs in process-log/ai-sessions/

Failure to disclose AI use when evidence of it is found during review will result in rejection.

Recommended AI Workflows

Not all AI workflows are equal. AIDER requires that your research process is documented and reproducible — this means we care about how you use AI, not just which model you use. The table below shows which types of AI tools produce the kind of process records we need.

We do not recommend any specific model or provider. Use whichever model works best for your research domain. For guidance on model capabilities, consult public benchmarks such as Chatbot Arena, Aider Code Editing Leaderboard, or SWE-bench.

Tool Type Examples Rating Auto Logs Edits Files Directly Process Reproducibility Notes
CLI agents Claude Code, Aider, Codex CLI, Copilot CLI Recommended ✓ ✓ ✓ Session transcripts are saved automatically. The agent reads, writes, and runs code inside your repo. The full development history is captured.
API / programmatic Any model via API, LangChain, custom scripts Recommended ✓ ✓ ✓ You control the pipeline. Prompts and responses can be logged programmatically. Ideal for batch experiments and systematic analysis.
AI-native IDEs Cursor, Windsurf, GitHub Copilot in VS Code Acceptable ~ ✓ ~ Edits files directly and some keep chat logs, but logging completeness varies. Export your chat history and include it in your process log.
Desktop apps Claude Desktop, ChatGPT Desktop Acceptable ~ ~ ~ Some can access files on your computer, which is better than web. But process logging is limited. Export conversations and include them manually.
Web chat interfaces ChatGPT web, Claude web, Gemini web Not Recommended ✗ ✗ ✗ Requires manual copy-paste between browser and your codebase. Slow, error-prone, and produces no automatic record. The development process is effectively lost.

Why This Matters

AIDER's core principle is that the research process is part of the contribution. When you use a CLI agent or API, the full conversation — every prompt, every code edit, every error and fix — is captured automatically. This is exactly the kind of process log that makes research reproducible and verifiable.

When you copy-paste between a web chat and your editor, that record is lost. You'd have to manually reconstruct it, which is tedious, incomplete, and defeats the purpose. We're not banning web chat — but if you use it, you're creating extra work for yourself and a weaker submission.

How to Submit

  1. Create your repository from the paper template
  2. Populate all required directories and complete the reproducibility checklist
  3. Open a submission issue
  4. Fill in the submission form with your repository link and details
  5. The automated reproducibility pipeline will run within minutes
  6. AI agents will pre-screen, audit claims, and check references automatically

After Submission

  • Automated check: The CI pipeline verifies repository structure, installs dependencies, and runs reproduce.sh. If this fails, you will be asked to fix issues before human review begins.
  • Review: AI agents perform pre-screening, claim verification, and reference auditing. An editor reviews the AI reports and the submission, then makes the editorial decision. Additional reviewers may be invited at the editor's discretion. All reviews are posted publicly.
  • Revision: If revisions are requested, update your repository and comment on the submission issue. There is no limit on revision rounds.
  • Acceptance: Your repository is forked into the AIDER GitHub organisation and assigned a Zenodo DOI. Publication is permanent — the DOI cannot be deleted and academic search engines will index the paper automatically. See our Ethics & Publication Permanence policy for details. You may withdraw your submission at any time before acceptance by commenting on the issue.
